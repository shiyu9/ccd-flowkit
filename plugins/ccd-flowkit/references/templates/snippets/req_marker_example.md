# 要件定義書の `**REQ-NNN**` 太字マーカー付き見出しの例示

## 対策する問題

`trace_scan.py` は要件定義書から `REQ-NNN` を検出する際に、**太字マーカー `**REQ-NNN**`**
（アスタリスク2つで囲まれた形式）を要求する。単なる見出し `### REQ-NNN` では検出されず、
req_set が空集合になる。すると総合試験計画のトレース行 `**トレース**: REQ-NNN` が全件
「参照する REQ-NNN が要件/設計に存在しない（dangling）」の high 検出になる。

**発生しやすいパターン**:
- 要件定義書に `### REQ-NNN タイトル` の見出しのみで太字マーカーが欠落 →
  総合試験計画の全 TEST 項目のトレース行が dangling 化して high 検出

1箇所の欠落が試験計画全件のトレースに波及するため、被害範囲は最大級。

## 正しい書き方

### ✅ 正しい例（見出し内に太字マーカー）

```markdown
### **REQ-001** タスクの追加

**概要**: ユーザーが CLI からタスクを追加できる
**受入条件**:
- AC1: `add <title>` で新規タスクが作成される
- AC2: title 空文字は拒否される
- AC3: 追加後 tasks.json に永続化される
- AC4: 追加時に一意の ID が採番される
```

見出しレベル（`##` / `###`）はプロジェクトの階層構造に合わせて選ぶ。太字マーカー `**REQ-NNN**`
の存在が必須で、レベルは trace_scan には影響しない。

### ❌ 間違った例（trace_scan が検出できない）

**見出しのみ・太字マーカーなし**:
```markdown
### REQ-001 タスクの追加
```

**平文の太字**（見出し外・trace_scan が要件見出しと認識しない）:
```markdown
**REQ-001** ユーザーが CLI からタスクを追加できる
```

## 太字マーカーの意味と背景

trace-conventions.md「要件の記法」節の規約:
- 見出し内に `**REQ-NNN**` を含めることで、
  - (a) trace_scan が要件定義書から REQ-NNN を機械的に抽出できる
  - (b) 目視でも太字で視認性が高い
- 平文の `**REQ-NNN**` 参照（例: 「REQ-001 の要件を満たす…」）は見出しでないため
  trace_scan は要件定義箇所として扱わない（散文の参照として扱う）

## 全REQ の網羅確認

要件定義書の全REQ に太字マーカーが付与されているか確認するには:

```bash
grep -c '^\*\*REQ-' docs/requirements/要件定義書.md          # ⚠ 見出しは頭に ### が付く
grep -cE '^#+\s+\*\*REQ-' docs/requirements/要件定義書.md    # ✅ 見出し行の太字マーカーを数える
```

期待される件数 = 要件定義書の REQ 総数。0件や不足があれば trace_scan で dangling が
発生する。
