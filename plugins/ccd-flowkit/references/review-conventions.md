# レビュー観点規約

レビュー担当エージェントは該当節の観点で確認し、各指摘に
「重要度(高/中/低) / 該当箇所 / 指摘 / 是正案」を付す。未是正の「高」が残る場合は不合格。

## レビューの真正性（必須・自己レビュー/捏造の禁止）
- **必ず別エージェントに委任する**。integrator（旧集約役）は自前の直接分析を観点サブエージェント
  の所見として記録してはならない（自己レビューは独立性が無く追認になる）。所見は state に保存された
  観点サブの結果だけを記録する。**フラット化モデルでは観点サブの起動は main
  （ディスパッチャ）が担う**——integrator は起動責務を持たず、state/review_results/&lt;NODE&gt;.jsonl
  を読んで統合・判定・記録・advance するだけ。main は各観点サブを**並列に起動する**（推奨: 単一メッセージで
  全観点 Task を同時呼出し）。フラット化モデルでは Task の呼び出し元が main であるため完了通知は必ず
  main に返り、旧集約役モデルで起きた「子孤児化」は構造的に発生しない（直列制約は集約役が Task 直後に
  停止する構造への対症療法であり、フラット化後は不要）。並列化により所要時間は最も重い観点の実行時間に短縮される。
  **不変条件**: サブエージェントは他のサブエージェントを起動しない（hook_check_subagent_launch が
  agent_type 付きの Task|Agent を無条件 deny する）。旧モデルの run_in_background 特例・逆委任個別
  ガード・「集約役の早期終了」対策は本不変条件で一括解消（集約役の階層構造による問題を構造的に解消）。
- **暫定記録の禁止**。「サブエージェントはバックグラウンドで並行起動中」「結果が届き次第追記」等、
  未確定のまま所見を記載しない。全サブエージェント結果が確定してからレビュー記録を生成する。
- **起動の正直な申告（正準宣言行）**。レビュー記録に1行 `**起動サブエージェント**: name1, name2, ...`
  を置き、**実際に起動した素のエージェント名だけ**をカンマ/読点区切りで列挙する。未起動を「起動済み」と
  書かない。review_authenticity_scan.py はこの宣言行を厳密にパースして起動ログ(_agent_invocations.jsonl)と
  突合し捏造を high 検出する（散文からの推測はしない。宣言行以外の DES-001 等のID言及は対象外）。
- **レビュー記録は別ファイルに置く**。レビュー記録は対象成果物の本体に書かず、専用ファイルに記す
  （設計=docs/design/設計レビュー記録.md、構築=docs/build/構築レビュー記録.md、試験計画=各フェーズの
  `*試験計画レビュー記録.md`）。設計書・計画書は最新状態のみを保持し、履歴・レビュー記録を本文に残さない
  （履歴の正本は Git。document-standards.md）。
- **レビュー記録内の例示表記**。「是正時の参考」等の説明目的で決定値トークン `[D:id=value]` や
  ID記法（`TEST-NNN` 等）を実参照でなく**例として**書く場合は、コードフェンス（\`\`\` で囲むブロック）で
  囲む。インラインバッククォート（`` `...` ``）や地の文への直書きで例示すると、ontology_scan.py /
  trace_scan.py が実参照と誤認し高指摘を誤発火させる（コードフェンス内は両スキャナとも走査除外対象。
  インラインバッククォートは実参照側でも使われうるため走査除外にしていない＝除外の強化ではなく、
  例示側の書き方を変えることで解決する）。
- **再レビューの実質化（3成分モデル）**。是正後の再レビューは次の3成分で行い、
  「対応状況=是正済み／再判定=PASS（根拠）」を記す。「未是正」のまま「再判定=—」で次フェーズへ送らない。
  - **中立の原則（毒は「主張」であって「機械diff」ではない）**: 是正者・仲介者の完了報告
    （何をどう直したか・「是正済み」「pass 可」等の結論）は受け取らない/従わない。一方、
    **レビューア自身が機械的に取得する変更情報**（`manage_review_baseline.py diff <NODE> <対象>`＝
    前回レビュー時点との unified diff）は是正者に改竄できない中立な事実であり、再レビューの一次材料にする。
  - **(a) 指摘消込**: 前回レビュー記録の指摘一覧（是正者の報告でなくレビュー記録が正本）を1件ずつ
    現在の成果物に当て、解消を独立判定する。
  - **(b) 回帰確認**: 機械 diff の変更箇所とその影響範囲（呼び出し元/依存先/同じ値の restatement）に
    全観点を当て、是正が他所に生んだリグレッション（新たな不整合・IF/例外/境界の破れ）を探す。
  - **(c) 決定論チェックは毎回全量**: スキャナ相当の形式確認（トレース行・宣言行・件数整合）は
    diff に関係なく全体に掛ける（全体網羅は機械が安価に担保する）。**試験計画レビュー
    （test-plan-convention-reviewer）では、この原則を文字通り実行する**: LLM が目視で全量再確認
    するのではなく、trace_scan.py / ontology_scan.py を実際に呼び出し出力を引用する
    （呼び出しのたびに規約ファイルを再読みして全文照合する高コストな目視
    チェックを、既存の安価な決定論スキャナへ委譲する）。**既知のトレードオフ**: この委譲のため
    test-plan-convention-reviewer に `Bash` を付与した（従来は Read/Glob/Grep のみ）。
    本プラグインの integrator エージェント（design/build/各test-plan-reviewer）は元々 Bash を保有し、
    上流エージェントが生成した文書を読んでコマンドを組み立てる同様の設計であるため、本変更は
    既存の権限モデルの範囲内の拡張である。個別エージェント単位でのBash allowlist機構
    （hook_check_commit_agent.py のような特定コマンド限定hook）の導入は将来の設計検討課題とする。
  - **全文再レビューは「初回」「baseline 無し」「変更が全体の3割超」のみ**（このとき初回と同じ全観点フル）。
  - **baseline の保存**: レビュー終了時（pass/high とも）に `manage_review_baseline.py save <NODE> <対象>` で
    今回レビューした状態を保存する（次回 diff の基準）。save は**当該レビューノードの担当レビュワーのみ**
    実行できる（hook が実行主体を検証。是正者が baseline を先回り上書きして diff を空に見せる偽装を
    機械的に防ぐ。diff/hash の参照は誰でも可）。
  - **軽量スキップ（試験計画レビュー限定）**: `manage_review_baseline.py save/diff <NODE>`
    には**対象成果物と上流文書の両方**を渡す（複数ファイル引数対応。上流ドリフトも同じ機構で機械検知する。
    初回は診断結果が `no_baseline` になり以下の条件を満たさないため、通常通り3.のフル/デルタ委任に進む
    ＝取得した diff 自体は無駄にならず「スキップ不可」の判定材料として使われる）。
    フラット化モデルでは **main が委任前に判定** する（integrator は起動しないため）。この diff で
    **対象・上流の両方が `status: "unchanged"`（変更率0.0%）、かつ前回レビュー記録に未是正の「高」
    指摘が0件**（＝review_round が前回で1にリセットされている＝収束規律の対象外）の場合に限り、main
    は観点サブエージェント（test-plan-convention-reviewer 等）の起動を全てスキップし、直接 integrator
    を起動できる（integrator は state に該当ラウンドのレコードが1件も無いことを検知し「軽量スキップ・
    観点サブ起動なし」の旨を記録する）。この判定は機械的事実（diff・前回レビュー記録）のみに基づき、
    内容の合否をLLMが新たに判断するものではないため中立の原則を破らない（diff 0.0% は「前回審査済み
    の内容と完全に同一」を意味し、再審査すべき新規内容が存在しないという論理的帰結）。適用対象は
    SYS/INT/UNIT_REVIEW の3ノード（試験計画レビュー）。他のレビューノード（DES_REVIEW/BLD_REVIEW等）には
    適用しない（対象ファイル1点のみのdiff運用のままでよい。試験計画レビューは呼び出し回数が
    突出して多いため優先適用）。
    **記録形式（review_authenticity_scan.py との整合が必須）**: `**起動サブエージェント**:` 宣言行に
    観点サブの名前を書いてはならない（`extract_claimed_agents()` が「実際に起動した」宣言として
    抽出し、起動ログと不一致＝捏造 high を誤発火させる）。宣言行は「（軽量スキップ・観点サブ起動なし）」
    等、エージェント名を含まない文言にする。**その代わり**、宣言行とは別の箇所（例:
    「**継続確認**: test-plan-convention-reviewer はスキップ — 対象・上流とも変更なし
    （diff出力を引用）・前回PASS維持のため未起動」）に、観点サブの素名を「スキップ」等の語の
    60文字以内に記す（`check_reviewer_coverage()` が宣言行以外も含めた記録全文からこの近接パターンを
    検索し、観点網羅を確認する。宣言行内への記載と宣言行外への記載を混同しないこと）。
    **diff出力は逐語で引用する**（security review指摘: `diff` サブコマンドは `save` と異なり
    実行主体を検証する hook が無く、状態機械の監査ログにも残らない。「diff引用」を単なる散文の主張に
    せず、`manage_review_baseline.py diff` の実際の標準出力（JSON部分の `status`/`change_ratio` を
    含む）を要約せずそのまま貼ることで、スキップ判定の根拠を事後に機械的（diffを再実行して突合）・
    人手いずれでも検証可能にする。要約や言い換えのみのスキップ記録は不十分＝未是正/差し戻しに準じる
    扱いとする）。
- **収束規律（ラウンド上限）**: レビューのラウンド数（ディスパッチャが委任時に伝える review_round）が
  **上限（正本: manage_flow_state.py の REVIEW_ROUND_ESCALATION_THRESHOLD＝現在3）を超える場合、
  advance せず「要ユーザー判断」を返す**（指摘の振動・仕様曖昧の兆候。ユーザーが免除/続行/方針変更を
  判断する）。上限超過のままの `advance <*_REVIEW> pass` は hook が機械的に block する。ユーザーが
  免除を承諾した場合のみ、ディスパッチャの再委任指示に従い免除注記（下記 waiver）を書いたうえで
  `advance <node> pass waiver` で通す（waiver は state の history に記録され事後検証される）。
  あわせて: **レビューアは1ラウンドで全観点の全指摘を出し切る**
  （再レビューでの新規指摘は回帰起因のみ正当。小出しはラウンドを浪費する）。**是正側は未是正の全指摘を
  1回の FIX で直し切ってから advance する**（1件ずつの逐次消化をしない）。
- **承認・判断の裏付け（無根拠の承認禁止）**。指摘を「ユーザー承認/判断」で合格化・解消する場合は、
  AskUserQuestion を実施しユーザーの明示回答を decisions.md/json に記録し、その参照を併記する。
  裏付け参照の無い「ユーザー承認済み」表記は認めない（review_authenticity_scan が high 検出する）。
- **ユーザー承諾による免除（waiver）の明示記録**。レビューが収束せずユーザーが「スルーする／
  この観点は免除する」と承諾した場合、レビュー記録に1行 `**免除**: <範囲> — ユーザー承諾（日付・
  decisions 参照）` を追記してから pass にする。免除注記なしで「FAIL/未是正の高が残る記録」のまま
  state だけ pass で進めない（後続監査が矛盾として誤検出するのを防ぐ）。免除の承諾自体は
  decisions.md/json に記録し参照する（上記「承認・判断の裏付け」と同じ扱い）。

## 設計レビュー観点
- 要件定義書の全要件が設計に追跡できるか（トレーサビリティ）
- 単一責務・疎結合になっているか
- エラー処理・例外系・境界条件が設計されているか
- 非機能要件（性能・セキュリティ・可用性）が反映されているか
- **アルゴリズム/式の論理的妥当性を境界で検証**: 設計の主張が、その式で本当に成り立つかを境界
  （最小・最大・空・全削除・全消去後の再操作）で確かめ、反例を探す（例: 「max(id)+1 は再利用しない」は
  最大IDを削除して再追加すると最大IDを再利用するため主張が偽）。主張と実装方式が
  境界で食い違えば高指摘。
- **要件由来の決定値方針の反映**: ID採番・削除後挙動などの決定値が要件と整合し、要件内・要件⇔設計で矛盾が
  無いか（全削除等の境界ケースの挙動が明記され両立不能でないか）。
- エビデンス（根拠）の妥当性: 全判断に `[E_n]` が付き未検証 `[E?]` が残っていないか、各 `[E_n]` の
  参照先（`{file}:{line}`/VRR/ud）が実在し主張を実際に支えているか（evidence-conventions.md）
- **DES間依存の意味整合・decisions来歴の妥当性**（DES_GATE から移管＝ゲート/レビュー責務分離）:
  依存先 DES の IF・引数・戻り値と矛盾なく使っているか。confirmed 決定の value が cited evidence
  （VRR/ud/codebase）と整合し、実装・試験結果と矛盾しないか（ontology_scan は参照の存在のみ見るため、
  値が実際に成立するかはレビューが担う）。矛盾は高指摘。
- **インターフェース（IF）連携の先読みチェック**: 設計変更が他コンポーネント・
  外部システムとのIF宣言（引数・戻り値・例外）に影響する場合、呼び出し元での例外・タイムアウト処理の
  波及と、下流の結合試験計画（INT_PLAN）が前提とする挙動への影響を先読みする。IF変更がINT側の
  再検討・再レビューを要する見込みなら、その旨を指摘（中）として明示する（DES_GATE⇔INT_REVIEW間の
  手戻り往復を、DES_REVIEW時点での先読みにより減らす）。

## 構築レビュー観点
- 設計書との整合（設計外の挙動が無いか）
- 命名・コーディング規約の遵守
- エラーハンドリングとログ
- 明白な脆弱性・リソースリーク・未処理の戻り値が無いか
- **設計-実装の事後整合**: 実装の是正が設計の記述（IF・例外方針・終了コード等）を超過/変更している場合、
  設計書の該当記述も更新されているか。実装だけ直して設計が古いまま（設計=未捕捉・実装=捕捉 等の乖離）に
  しない（設計=未捕捉・実装=捕捉のような食い違いを残さない）。
- **設計に無い副産物は高指摘**: 実装が設計に記載の無いファイル・関数・永続化・振る舞いを持つ場合（例:
  設計に無い `.todo_next_id` のような永続化ファイル）は高指摘とし、設計反映まで合格にしない（条件付き合格で通さない）。

## 試験計画レビュー観点
- 上流成果物の全項目に対応する試験項目があるか（網羅性）
- 正常系・異常系・境界値が揃っているか
- 各試験項目に合否判定基準が定義されているか
- 前提条件・試験データ・手順が再現可能か
- **正準トレース行（trace-conventions.md）**: 各試験項目が `TEST-NNN` 見出し＋`**トレース**:` 行を持ち、
  レベル相応の上流型（総合→REQ / 結合・単体→DES）を含み、参照先が実在するか。欠落・dangling・旧記法
  （`#NNN`）は trace_scan が high で止めるため、レビューで先に是正させる（高ゲートの記法崩れ誤発火を防ぐ）。
- **決定値の矛盾（REQ_GATE の意味整合から移管）**: 期待値が上流（要件・設計・decisions.json の確定値）と
  食い違っていないか。要件内で決定値方針が矛盾（例: 『全削除後ID=1』と『削除ID再利用禁止』の衝突）して
  いないか。決定由来の期待値が `[D:id=value]` トークンでなく散文直書きになっていないか。矛盾は高指摘
  （偽陰性・偽陽性の温床となるため）。
- **入力例示の具体性**: 試験項目の入力・手順に「コマンド」「入力」「操作」等の総称語が
  残っていないか。要件文の総称語をそのまま転記すると、具体的なコマンド種別を誤って選んでしまう
  （例: 読み出し専用の `list` を誤選択）。総称語のまま残っていれば指摘（中）。

## 試験結果レビュー観点（実行証跡・ごまかし防止／test-conventions.md）
- 実績欄に実行証跡（コマンド・出力抜粋・終了コード）があるか。◯/PASS の表だけなら指摘（高）
- テンプレート未展開（`{datetime.now()...}` 等のプレースホルダ残存）が無いか。あれば指摘（高）
- スキップを PASS 計上していないか（SKIP は独立ステータスで理由付き）。あれば指摘（高）
- FAIL を読み替えて合格化していないか。あれば指摘（高）
- サマリー合計と個別項目数が整合しているか。不整合は指摘（中）
- 是正後の更新漏れが無いか：サマリーと集計表・見出しと本文・上流文書(設計/試験計画)との restatement で
  件数/判定/値が一致するか（是正後に片方だけ更新した陳腐化）。不一致は指摘（高）

## 判断エスカレーションの妥当性（全レビュー共通）
担当が返した「要ユーザー判断」やユーザーへの質問が escalation-conventions.md に沿うか確認する。
- 質問すべき6基準のいずれにも該当しないのに質問している → 「質問妥当性: 不当」（中）
- 基準番号の明示がない／書式（状況・不明点・影響・推奨）が欠ける → 指摘（中）
- 6基準に該当するのに推測で進めている（質問すべきを質問していない）→ 指摘（高）
