ccd-flowkit 開発フロー フローチャート v3-current（integrator モデル）

フラット化モデル・現行実装の正本。旧集約役モデル（main → 集約役 → 観点サブ×N の階層構造）は、Claude Code の Task 非同期モデルにおいて子完了通知が親でなく main へ届く仕様と不整合であり、集約役の再委任のたびに fresh インスタンスが立ち上がり過去の観点結果を失う構造欠陥を持っていた。フラット化モデルはこれをフラットな三段構成（main が観点サブを直接並列起動→ state に結果保存→ integrator が state を読んで統合＋判定）に解消し、サブ→サブの起動を不変条件として禁止する。 対象は 5 レビューノード全て（DES/SYS/INT/BLD/UNIT_REVIEW）。状態機械（図1のノード位相）は不変で、変わったのは各レビューノードの内部フローとフック配置のみ。

フラット化の一言: 集約役（build-reviewer 等）が「観点サブを起動して結果を統合する」多責務エージェントだったのを、 「main が観点サブを直列起動→結果を state に保存→integrator（旧集約役）が state を読んで統合＋判定」の3段構成に分離した。 サブ→サブの起動は普遍的に禁止し、階層的呼び出しの再発を構造的に防ぐ（hook_check_subagent_launch が agent_type 付き Task|Agent を無条件 deny）。

フラット化モデルの構成要素

  1. state/review_results/<NODE>.jsonl: 観点サブの結果を保存するファイル（main が Bash append で書込、integrator が Bash で読取）。
  2. integrator（レビュワー）: state を読んで統合＋判定＋advance するのみ（Task ツールを持たない・観点サブの起動責務なし）。5レビューノード（SYS/DES/INT/BLD/UNIT_REVIEW）全てで採用。
  3. reviewers.json のキー体系: `sys_review` / `des_review` / `int_review` / `bld_review` / `unit_review`（ノード名の小文字形と一致）。
  4. check_rerun_limit.py: integrator の rerun 要求を機械的に上限判定（RERUN_LIMIT_PER_ROUND=2）。超過時は main が AskUserQuestion でユーザーに諮る（強制収束はしない）。
  5. hook_check_subagent_launch: 「agent_type 付き Task|Agent は無条件 deny」に一般化（サブ→サブ起動を不変条件として禁止）。
  6. hook_check_state_write.evaluate_review_results: state/review_results/*.jsonl への Write/Edit を main 専用に限定（Bash 側の evaluate_rerun_actor と対称）。
  7. hook_check_advance_actor.evaluate_rerun_actor: rerun / review_results 書込の実行主体を main に限定。
リスクと軽減策: フラット化は状態機械を変えないが、main の SKILL.md 制御ロジックが増える。 main のプロンプト複雑化を機械化スクリプト（check_rerun_limit.py・将来的な next_reviewer.py）で吸収する。 比較の基盤（scorecard 6観点）は不変。
