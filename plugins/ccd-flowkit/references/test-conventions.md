# 試験規約

## 試験計画の階層
| 計画 | 立案フェーズ | 目的 |
|---|---|---|
| 総合試験計画 | 要件定義 | 要件を満たすかを利用者視点で確認する観点を定義 |
| 結合試験計画 | 設計 | モジュール間連携・IFを確認する観点を定義 |
| 単体試験計画 | 構築 | 関数/モジュール単体の正常・異常・境界を定義 |

## 試験項目の必須要素
試験ID / 目的 / 前提条件 / 入力 / 手順 / 期待結果 / 合否判定基準 / 紐づく上流項目

- **試験IDと正準トレース行**（trace-conventions.md）: 各試験項目の見出しに正準ID `TEST-NNN`（3桁固定）を
  置き、見出し直下に1行 `**トレース**: <上流ID群>` を記す。総合は REQ、結合・単体は DES を1つ以上含める
  （関連する他種IDも可）。trace_scan.py がこの行を収穫し、参照整合・型・カバレッジを high で検査する。
  ```markdown
  ### TEST-001 add コマンド正常系
  **トレース**: DES-002, REQ-001
  ```

## 試験結果の必須要素
試験ID / 実施日 / 実績 / 合否 / 不具合ID（NG時）/ 再試験結果

- **不具合IDは試験結果横断で一意**: 不具合IDは単体/結合/総合をまたいで一意に採番する（レベル接頭辞
  UT-/IT-/ST- 等、または通し番号）。総合BUG-001と単体BUG-001のように別スコープで同番号を使わない
  （追跡性を保つ）。

試験結果（docs/test/*試験結果.md）は**各 tester が書く**。オーケストレーターは直編集せず該当 tester に
委任する（数値訂正・是正も tester への差し戻しで行う）。委任ガード hook が、agent_type の無い
メインセッションからの試験結果の直接書込をブロックする。

## 実行証跡の必須化（ごまかし防止）
- **実際に実行して記録する**。実績欄には実行コマンド・標準出力/標準エラーの抜粋・終了コードなど
  「実行した事実」を示す生の証跡を残す。◯/PASS の表だけでは不可。
- **テンプレート未展開の禁止**。`{datetime.now()...}` のようなプレースホルダや雛形文字列を
  そのまま残さない（実行・整形した結果でないことの証拠になる）。実施日は実際の日付を書く。
- **実施環境を正確に一貫して記録する**。OS・言語バージョン・作業ディレクトリは実際に実行した
  環境を書く（例: メタ表は Windows 11 なのに本文の作業ディレクトリが /tmp（Linux）等の食い違いを
  作らない）。値は推測で書かず、実行時の事実を記す。
- **作業ファイルを成果物に残さない**。被試験プログラムが実行時に書き出すデータファイル（例: アプリの
  tasks.json）や試験用の作業ディレクトリは「作業ファイル」であり、成果物（試験結果・レビュー記録など
  docs/ 配下の文書、src/ のソース）とは別物。作業ファイルは一時領域で実行し、成果物ツリー（src/・docs/）に
  混入させない。混入したら片付ける。例外は次項の**機械記録**（docs/test/evidence/ に置く証跡成果物）のみ。
- **ハーネス一括実行は機械記録を残し、宣言と突合可能にする**: 単体試験のようにハーネスで一括実行する場合、
  ハーネスが出力する機械可読の結果記録（JSON。各件に `status`/`result` 等で PASS/FAIL/SKIP）を
  `docs/test/evidence/` に保存し、結果文書に正準行 `**機械記録**: evidence/<ファイル名>` で参照する。
  **結果文書の合否宣言の実数と機械記録の実数は一致必須**（result_scan.py が突合し、不一致・参照先不在は
  high、単体試験結果に機械記録行が無ければ medium）。ハーネス自体が不具合を起こした場合（クラッシュ・
  文字化け等）は、**修正→再実行の顛末（何が壊れ、どう直し、最終記録はどの実行か）を結果文書に記録する**。
  壊れた実行の記録を残したまま宣言だけ PASS にしない（機械記録と宣言の乖離を防ぐ）。
- **合否ステータスの規律**: 合否は PASS / FAIL / SKIP の3値のみを記録する。
  - **合否宣言の正準形式**: 各試験項目の最終合否は1行 `**合否**: PASS`（コロン必須。PASS/FAIL/SKIP）で
    宣言する。result_scan.py はこの宣言形式を数えてサマリーと突合する——テーブルのセルだけの合否記載は
    個別実数として集計されず、件数不一致の high を誘発するため。
    **派生ラベルは経過記録に使ってよい**: 「初回合否:」「再試験合否:」など「合否」の直前に漢字が続く
    ラベルは最終合否として**集計されない**（二重計上を防ぐためスキャナ側で除外済み）。
    最終判定は素の `**合否**:` を1項目につき1行だけ書く。
  - **判定語彙の統一**: 「条件付きPASS」「条件付き合格」は使用しない。条件付きで合格とする
    場合は合否=PASS とし、条件や留意事項は備考欄に記載する（result_scan.py が正しく集計するため）。
  - **期待結果は全条件成立を要する**: 期待結果に複数条件があるとき、一つでも実績が不成立（✗）なら
    PASS にしない。FAIL とするか、計画（期待結果）が誤りなら計画を是正してから再試験する。一部条件の
    成立を理由に PASS へ寄せない。
  - **「環境要因」での無罪化には機械証跡を要する**: 期待した終了コード/出力と実績が食い違うのを
    『試験環境要因・実装は正常』として実装無罪化・PASS 寄せする場合は、機械再実行の証跡（実行コマンド・
    標準出力/標準エラー抜粋・終了コード）を必ず添付する。証跡の無い『環境要因』での合格化は認めない。
    exit code の期待不一致は、証跡で環境起因と示せない限り FAIL とする。
  - **SKIP の無罪化にも代替経路の機械再実行を要する**: I/Oエラー系（書込権限拒否・OSError 等）を
    「環境で強制できない」として SKIP・実装無罪化する場合、コードを読んだだけの無罪化は認めない。
    その実行環境（例: Windows）で**実際に発火させられる代替エラーパス**（読取専用ファイル・不正パス・
    ディスク不能など）で機械再実行し、実装のエラーハンドリング（exit コード・stderr）が期待どおり
    働く証跡を添付する。代替経路でも発火不能な場合のみ、その事実（試みたコマンドと結果）を SKIP 理由に記す。
  - スキップした試験を PASS として計上しない（環境制約等で未実行なら SKIP と明記し理由を書く）。
  - FAIL を「テスト側の制限」等の理由で実装合格に読み替えない。FAIL は FAIL として残し、
    原因（実装/設計/要件/試験のどれ起因か）と対応を記す。
  - **原因が試験計画側なら計画を是正してから再評価する**: FAIL の根本原因が実装ではなく**試験計画の
    期待値誤り**（上流の要件/設計と食い違う期待値を書いていた等）と判定された場合、FAIL のまま放置・
    「部分カバレッジ」への格下げで済ませない。試験計画（承認済み成果物）の期待値を是正し、当該試験を
    再評価（PASS/FAIL を付け直す）してからゲートを通す。実装が正で計画が誤りなら偽陰性 FAIL を残さない。
- **件数整合**: サマリーの合計件数と個別項目数を一致させる。

## 是正・再試験後の更新漏れ防止
ある事実（件数・判定・exit code 等の値）を複数箇所に記している場合、片方だけ是正して他方が
陳腐化すると文書内矛盾になる。是正・再試験のたびに次を守る。
- **全 restatement を更新**: サマリー・分類別集計表・各項目の合否・見出し・本文・上流文書への参照を
  すべて新しい結果に更新する（例: BUG 是正後にサマリーは FAIL 0 に直したが集計表が FAIL 2 のまま、を作らない）。
- **自己整合の確認**: 文書内（サマリー vs 集計表、見出し vs 本文）・文書間（試験結果 vs 設計/試験計画の値）で
  件数・判定・値が一致することを確認する。件数の自己整合は result_scan.py が決定論的に検査する。
- **決定由来の値は正本から引く（規約: ontology.md）**: 終了コード・真偽・列挙の選択肢など
  `docs/design/decisions.json` に正本がある値を試験計画・結果に書くときは、直書きせず参照トークン
  `[D:id=value]`（例 `[D:UD-06=1]`）で引く。決定が後で変わってもトークンと正本の食い違いを
  ontology_scan.py が検出する（複製値の陳腐化防止）。複雑値（フォーマット文字列等）は直書きでよい。
- **是正担当は advance 前に result_scan を自分で再実行し high=0 を確認する**: 是正の書き込みが
  別のパターンに抵触する連鎖を、ゲートへ回す前に自分で断ち切る。
  委任プロンプトに渡された絶対パスの `scripts/result_scan.py` を Bash で実行すればよい。
- **合否宣言形の文字列の例示・引用はコードフェンスで囲む**: 是正記録・経緯説明で旧記録の
  「合否: FAIL」等を引用するときは ``` フェンス（またはインラインコード）内に置く。
  result_scan.py はフェンス・インラインコード内の宣言形・機械記録行を集計から除外する
  （地の文に書くと実宣言として二重計上され件数不一致 high を誤発報する）。

## 回帰後の再試験の実体化（形式だけの再通過の禁止）
RCA からの回帰（impl/design/req/test）後、状態機械は以降の全テストレベル（T_UNIT→T_INT→T_SYS）を
再通過させる。このとき各 tester は次のいずれかを行う。**既存の結果文書を Read しただけで advance pass
する「無言の形式再走」は禁止**（実行実体の無い再 pass を防ぐ）。
- **src に変更がある**（前回結果の時点とハッシュ不一致）: 影響する試験を機械的に再実行し、結果文書に
  再試験記録（実施日・コマンド・出力抜粋・終了コード）を追記して版を更新する。
- **src が無変更**（ハッシュ一致を自分で確認。例: `git hash-object src/*.py` や sha256 比較）: 再実行を
  免除できるが、結果文書に「**再試験免除**: src 無変更（ハッシュ値・確認日）」を明示記録してから
  advance pass する。免除の記録なしに pass しない。

## 出力エンコーディングの前提を統一する（環境依存FAILの判定分裂防止）
被試験プログラムの標準出力に非ASCII（絵文字・✓ U+2713・日本語）が含まれる場合、Windows の cp932
コンソールでは UnicodeEncodeError でクラッシュしうる。**結合と総合で同一プロダクトの同一挙動が、
一方は素のcp932でFAIL・他方は errors='replace' でPASS のように判定が割れる事態を避ける**。
- 各試験の実施環境（OS・コンソールエンコーディング・PYTHONIOENCODING・subprocess の text/encoding/errors 方針）を
  試験結果に明記し、層をまたいで揃える。
- 環境依存FAILは「実装の堅牢性課題」と「試験環境の制約」を切り分けて記す。実装側の是正余地（出力ストリームの
  encoding 明示等）があれば指摘し、再試験任せにしない。同一挙動を層によって PASS/FAIL に振り替えない。

## チェック観点
- 正常系: 代表値で期待結果を満たすか
- 異常系: 不正入力・例外で適切に失敗・回復するか
- 境界値: 最小/最大/閾値前後で正しいか
- **失敗境界を突く境界データ網羅**: 最小・最大・空・全削除・全消去後の再操作など、実装が壊れやすい境界を
  必ず含める。特にID/採番・削除系は「**最大IDの削除→再追加**」（再利用が起きる境界）や「全削除後の再追加」を
  試験項目に入れる（最小IDの削除では再利用の失敗境界を外すため）。
