# 試験項目の `**トレース**:` 行の例示

## 対策する問題

`trace_scan.py` は試験計画の各試験項目に `**トレース**: <上流ID>` 行が存在することを
要求する。欠落・書式ミス・旧記法（`#NNN` 等）の使用で high 検出。

**総合試験レベル別の上流ID型**:
- **総合試験計画** (`docs/requirements/総合試験計画.md`): 上流は REQ（`REQ-NNN`）
- **結合試験計画** (`docs/design/結合試験計画.md`): 上流は DES（`DES-NNN`）
- **単体試験計画** (`docs/build/単体試験計画.md`): 上流は DES（`DES-NNN`）

## 正しい書き方

### ✅ 正しい例（総合試験・上流 REQ）

```markdown
### **TEST-001** タスク追加の正常系

**トレース**: REQ-001
**分類**: 正常
**前提条件**: tasks.json が存在しない状態
**手順**: `python src/todo.py add "買い物"` を実行
**期待結果**: 終了コード 0、tasks.json に id=1 のタスクが記録される
**合否判定基準**: tasks.json のハッシュが期待値と一致
```

### ✅ 複数の上流ID（カンマ区切り）

```markdown
**トレース**: REQ-011, REQ-012
```

### ✅ 正しい例（結合/単体試験・上流 DES）

```markdown
### **TEST-025** cmd_add と save_tasks の連携

**トレース**: DES-001, DES-005
**分類**: 正常
...
```

### ❌ 間違った例

**旧記法（`#NNN`）** — trace_scan が旧形式として扱わない:
```markdown
**トレース**: #001
```

**上流ID型のミスマッチ** — 総合試験に DES を書く / 結合試験に REQ を書く:
```markdown
### **TEST-001** タスク追加の正常系
**トレース**: DES-001                                  # ❌ 総合試験は REQ を書く
```

**太字マーカーなしのトレース行** — trace_scan がトレース行として認識しない:
```markdown
トレース: REQ-001
```

## 分類・前提条件・手順の書式

trace-conventions.md「試験項目の書式」節の規約:

- **分類**: `正常` / `異常` / `境界` のいずれか（それ以外の値は非推奨）
- **前提条件**: 具体的な状態を書く。「初期状態」等の抽象語は避ける
- **手順**: 具体的コマンドを書く。**総称語「コマンド」「入力」「操作」は禁止**
  （総称語のままだと実施者が読み出し専用の list を誤選択する等の実行時齟齬を招く）

## 上流ID実在の担保

trace_scan.py は上流ID が要件（REQ）/設計（DES）に存在することを検査する。存在しない
ID を書くと dangling high。上流未確定の時点で試験計画を書き始めるなら、上流側を先に
確定させるか、`<未定>` を明示して該当 TEST 項目を dangling として認識させる。
