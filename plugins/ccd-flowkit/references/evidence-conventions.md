# エビデンス（根拠）規約

設計書のすべての判断・命題に根拠（エビデンス）参照を付け、未裏付けの判断をゼロにする。
トレーサビリティ（要件→設計→コード→試験の接続）とは別軸で、各設計記述が検証された事実に
立脚することを担保する。

## 参照ラベル
- 確定: `[E_n]`（n は設計書ローカル採番、1始まり）。本文の判断の直後に置く。
- 未検証（一時）: `[E?:type:topic]`。designer が第1ラウンドで撒き、verifier が解決して
  `[E_n]` に置換する。**確定提出の設計書に `[E?]` が残っていてはならない**。
  `topic` は `^[a-z0-9_]{1,64}$`（小文字英数＋アンダースコア）に限定する（VRR/decisions の
  ファイル名に使うため。path traversal 防止）。

## 4種のエビデンス（type）
- **codebase**: 既存・周辺コードの事実。`{file}:{line}`。designer が Read で確認して書く（記憶ベース禁止）。
- **spec**: 外部仕様（ライブラリ・フレームワーク・API の挙動）。当面はバンドル/既知の仕様メモを
  引用し、不確実なら ud に倒す（ライブ fetch は将来。SSRF 対策が要るため MVP では持たない）。
- **vrr**: 実機検証記録（Verification Run Record）。verifier が Bash で検証し
  `docs/design/vrr/<id>.md` に記録、設計書はそれを引用する。
- **ud**: ユーザー確定事項。verifier が質問素案を作り、オーケストレーターが AskUserQuestion で
  確認、回答を `docs/design/decisions.md` に記録して引用する。確定値のうち下流が値として参照する
  スカラ（終了コード等）は、正本を `docs/design/decisions.json` に持ち decisions.md では復唱しない
  （値の二重持ち回避。詳細は ontology.md）。

## エビデンス一覧（設計書末尾）
確定した各 `[E_n]` を1行で定義する。書式は `[E_n] 型: 出典`（type は codebase/spec/vrr/ud）。
直後の「型＋コロン」が定義の目印であり、本文中の参照（`[E_n]` 単独）と区別される。例:
```
[E1] codebase: src/auth.py:10
[E2] vrr: docs/design/vrr/subprocess_encoding.md
```
本文の各 `[E_n]` 参照は、必ずこの一覧に定義を持つこと。エビデンス一覧の定義行は本文として書く
（`<pre>`/`<code>` ブロックの中に入れない。決定論スキャナはコードブロックを除外して判定するため）。

## 検討したアプローチ（設計書）
主要な設計判断は、採用前に候補を表で比較して残す（ID／アプローチ／利点／欠点／推奨度＋採用根拠 `[E_n]`）。
軽微で候補が自明な場合は「該当なし（理由: 1行）」を明示する。

## mark-then-verify（設計フローに組み込む）
1. **designer 第1ラウンド**: 各判断に `[E_n]`（codebase 等で確認済み）か `[E?]`（未検証）を付ける。
2. **verifier**: `[E?]` を解決する（codebase=Read、vrr=Bash 検証、ud=質問素案）。
3. **designer 第2ラウンド**: 検証結果を反映して `[E?]` を `[E_n]` に置換し確定する。

## チェック（フェーズゲートで実行）
- 決定論（`scripts/evidence_scan.py`）: `[E?]` 残存、未定義参照、未使用定義。high で fail-closed。
- 内容の妥当性（参照先が本当に主張を支えているか）は design-reviewer が見る（AI 主導）。
