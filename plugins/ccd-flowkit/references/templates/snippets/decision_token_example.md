# 決定参照トークン `[D:id=value]` の例示

## 対策する問題

`ontology_scan.py` は文書全体を無差別走査して `[D:id=...]` パターンを実参照とみなし、
`decisions.json` の台帳と突合する。台帳に無い ID は dangling として high 検出される。
このため「例示」として文字列を書きたい場合でも、地の文やインラインバッククォート
（`` `[D:x=1]` ``）だけでは実参照と誤認される。

**発生しやすいパターン**:
- 総合試験計画内の説明文中の例示がインラインバッククォートのみでコードフェンス外
- レビュー記録内の例示が同様（regression しやすい）
- 単体試験計画内の例示が同様

この記法の誤用は書式起因 high の主要因の一つ。

## 正しい書き方（コードフェンス必須）

例示として `[D:id=value]` を書く場合、必ずコードフェンス（```` ``` ````）で囲む。
ontology_scan.py はコードフェンス内を走査除外するため、誤検知が発生しない。

### ✅ 正しい例（コードフェンスで囲む）

決定判断由来の値を参照する場合、以下の形式を使う:

```
[D:default_limit=10]
[D:timeout_seconds=30]
[D:id_reuse_policy=disallow]
```

（フェンス内は例示。実参照は文中に直接書く形になる。）

### ❌ 間違った例（誤検知の温床）

**インラインバッククォートのみ**（スキャナが実参照と誤認）:
> 決定参照トークン `[D:id=value]` 形式で参照する。

**地の文への直書き**（同上）:
> 決定参照トークン [D:id=value] 形式で参照する。

## 実参照の書き方

実際の参照は文中に直接書いてよい（実在する `decisions.json` の ID を使う限り）:

> このAPIのタイムアウトは [D:api_timeout=30] とする。

## テスト・レビュー記録内で「例示」を書く場合の注意

test-plan-convention-reviewer や integrator のレビュー記録内で「トークン記法の誤用を
指摘する」文脈で `[D:id=value]` を書くこともよくあるが、その場合も必ずコードフェンスで囲む。
指摘対象の実物を引用する場合も同様（コードフェンス内なら安全）:

```
（対象成果物の18行目の記述）
    決定参照トークン `[D:id=value]` 形式で ...
                    ~~~~~~~~~~~~~~~
    ↑ このインラインバッククォート表記はスキャナ誤検知を招く
```
